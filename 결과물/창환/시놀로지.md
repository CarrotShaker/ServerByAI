
# 시놀로지 서버 구성 뜯어보기

## 1. 서버 환경구성 (docker)

**질문:** Docker에 대해서 자세히 설명해봐. Docker는 요즘 서버의 필수요소야? 서버 환경 자체를 묶어주는 하나의 묶음을 구성해주는 거로 인지했는데 이식성을 제외하고 어떤 효과를 추가적으로 볼 수 있어? 나는 "도커 컨테이너가 하나의 VM인가?" 이런 정도의 느낌을 받은 아무것도 모르는 개발자야

**답변:**

#### VM vs Docker 구조 차이 (핵심)

"VM 같은 느낌"이라고 했는데, 방향은 맞는데 구조가 완전히 다릅니다.

```
[ VM 구조 ]
---------------------------------
|  앱A  |  앱B  |  앱C          |
|-------|-------|---------------|
| OS    | OS    | OS            |  ← 각각 OS 통째로 들어감
|-------|-------|---------------|
|     Hypervisor (가상화 레이어) |
|-------------------------------|
|         실제 OS (Host)        |
|-------------------------------|
|           하드웨어             |
---------------------------------

[ Docker 구조 ]
---------------------------------
|  앱A  |  앱B  |  앱C          |
|-------|-------|---------------|
| 컨테이너A | 컨테이너B | 컨테이너C |  ← OS 없이 프로세스만
|-------------------------------|
|        Docker Engine          |
|-------------------------------|
|         실제 OS (Host)        |  ← OS 하나 공유
|-------------------------------|
|           하드웨어             |
---------------------------------
```

**VM은 OS를 통째로 올리는 것, Docker는 OS를 공유하고 프로세스만 격리하는 것**

---

#### 컨테이너가 뭔가요?

컨테이너 하나 = **앱 + 그 앱이 동작하는 데 필요한 모든 것**을 묶은 실행 단위입니다.

```
컨테이너 하나 안에:
  - 앱 코드
  - 런타임 (Node, Java, Python 등)
  - 라이브러리 및 의존성
  - 환경변수, 설정파일
```

이 컨테이너는 **호스트 OS와 완전히 격리된 프로세스**로 실행됩니다.

---

#### 이식성 외 추가 효과

**2. 환경 오염 방지**
```
// 예시: Node.js 버전 충돌 문제
앱A → Node 18 필요
앱B → Node 14 필요

Docker 없이 → 같은 서버에 두 버전 공존이 매우 까다로움
Docker 있으면 → 각 컨테이너가 완전히 다른 버전 격리해서 사용
```

**3. 의존성 고정 (재현 가능성)**
```dockerfile
# 이 파일 하나면 어디서든 동일한 환경 재현
FROM node:18
WORKDIR /app
COPY package.json .
RUN npm install
COPY . .
CMD ["node", "index.js"]
```
"내 로컬에선 되는데 서버에선 안 돼요" 문제 원천 차단

**4. 롤백이 쉬움**
```
v1.0 컨테이너 → v1.1 컨테이너로 교체
문제 발생 시 → v1.0 컨테이너로 즉시 복구
```

**5. 프로세스 격리 (보안)**
- 컨테이너A가 해킹당해도 컨테이너B, 호스트 OS에 영향 제한적
- 각 컨테이너별로 네트워크, 파일시스템 분리

---

#### 요즘 서버의 필수요소인가?

**필수는 아니지만 사실상 표준입니다.**

| 상황 | Docker 필요성 |
|------|-------------|
| 토이프로젝트 혼자 | 없어도 됨 |
| 시놀로지 서버 운영 | **강력 추천** (관리 편의성) |
| 팀 개발 | 거의 필수 |
| 클라우드 배포 (AWS 등) | 사실상 필수 |
| 마이크로서비스 | 필수 |

---

#### 시놀로지 기준으로 왜 편한가?

```
시놀로지에서 Docker 없이:
  → PostgreSQL 직접 설치 → 설정 파일 직접 수정 → 버전 관리 직접

시놀로지에서 Docker 있으면:
  → docker-compose.yml 파일 하나 작성
  → docker compose up 명령어 하나
  → PostgreSQL + API서버 + 기타 다 한번에 뜸
```

## 2. 백엔드 프레임워크 선택 (Spring / Ktor)


**Q1. 백엔드 프레임워크는 그냥 기능 일부를 제공하는 거 아냐? 어떤 기능들을 제공하는데? 안드로이드 프레임워크는 안드로이드의 모든 걸 제공하는데 서버 프레임워크는 어떤 걸 제공하는지 모르겠어. 각각 프레임워크마다 제공하는 범위도 다른가? Spring이나 Ktor 위주로 설명해봐**

**A1.**

안드로이드 프레임워크가 제공하는 것들:
```
- Activity/Fragment 생명주기 관리
- Intent로 화면 간 이동
- Context로 시스템 자원 접근
- RecyclerView, Compose 등 UI 컴포넌트
- Room, SharedPreferences 등 데이터 저장
```

서버 프레임워크도 비슷한 구조입니다. 다만 "화면" 대신 **"HTTP 요청/응답"** 이 중심입니다.

#### 서버 프레임워크가 제공하는 공통 기능

| 기능 | 설명 | 안드로이드 비유 |
|------|------|--------------|
| **라우팅** | `/api/users` GET 요청을 어떤 함수가 처리할지 연결 | Intent Filter |
| **요청 파싱** | JSON Body, 쿼리 파라미터 자동 파싱 | Intent extras |
| **응답 직렬화** | 객체 → JSON 자동 변환 | Gson/Moshi |
| **미들웨어** | 요청 전/후 공통 처리 (로깅, 인증 등) | OkHttp Interceptor |
| **의존성 주입** | 객체 생성/관리 | Hilt/Koin |
| **에러 핸들링** | 예외 발생 시 응답 처리 | 전역 Exception Handler |

#### Spring Boot vs Ktor 제공 범위 차이

**Spring Boot - 풀스택 프레임워크**
```
제공 범위: 거의 모든 것
  - HTTP 서버
  - DB 연동 (JPA, JDBC)
  - 인증/보안 (Spring Security)
  - 캐시, 메시지 큐, 스케줄러
  - 트랜잭션 관리
  - 테스트 지원

특징: 어노테이션 기반, 자동 설정(Auto Configuration)
      "설정보다 관례" → 알아서 다 해줌
      대신 무겁고, 내부 동작 모르면 블랙박스처럼 느껴짐
```

```kotlin
// Spring - 어노테이션이 다 처리해줌
@RestController
@RequestMapping("/api")
class UserController(
    private val userService: UserService  // DI 자동 주입
) {
    @GetMapping("/users/{id}")
    fun getUser(@PathVariable id: Long): UserResponse {
        return userService.findById(id)  // 리턴하면 JSON 자동 변환
    }
}
```

**Ktor - 경량 프레임워크**
```
제공 범위: HTTP 레이어 위주
  - HTTP 서버/클라이언트
  - 라우팅
  - 직렬화 (kotlinx.serialization)
  - 인증 플러그인

특징: 코루틴 기반, 명시적 설정
      필요한 것만 플러그인으로 추가
      가볍고 투명하지만, 직접 구성해야 하는 것이 많음
```

```kotlin
// Ktor - 명시적으로 다 선언해야 함
fun Application.configureRouting() {
    routing {
        get("/api/users/{id}") {
            val id = call.parameters["id"]?.toLong()
            val user = userService.findById(id!!)
            call.respond(user)  // 직접 응답 처리
        }
    }
}
```

#### 범위 차이 한눈에

```
Spring Boot  ████████████████████  (풀스택, 거의 모든 것 내장)
Ktor         ████████              (HTTP 레이어 + 플러그인 방식)
Express(JS)  ██████                (최소한만, 나머지 직접)
```

#### 안드로이드 개발자 입장에서 추천

- **Ktor** → Kotlin 코루틴 그대로 씀, 코드 스타일이 안드로이드와 유사, 토이프로젝트에 적합
- **Spring Boot** → 취업/실무 기준 압도적 점유율, 배워두면 범용성 높음

---

**Q2. 스프링도 코틀린으로 할 수 있잖아. 거기에 장벽이 있어?**

**A2.**

**쓸 수 있습니다.** Spring 공식적으로 Kotlin 지원하고, 실무에서도 많이 씁니다.

다만 장벽이 있는 이유는 **Kotlin 문법 문제가 아니라 Spring 자체의 개념 때문**입니다.

Spring이 어려운 이유 - 내부적으로 Java 생태계의 역사가 그대로 쌓여있습니다:
```
배워야 하는 개념들:
  - IoC / DI 컨테이너 동작 원리
  - Bean 생명주기
  - AOP (관점 지향 프로그래밍)
  - JPA / Hibernate (ORM)
  - Spring Security 구조
  - 트랜잭션 관리
  - Auto Configuration 동작 방식
```

이것들이 내부에서 **어노테이션 하나로 마법처럼 동작**하다 보니, 문제가 생겼을 때 원인을 모르면 디버깅이 매우 힘듭니다.

```kotlin
// Ktor - 읽히는 대로 동작
routing {
    get("/users") {
        call.respond(userService.getAll())
    }
}

// Spring - 어노테이션이 뒤에서 다 처리
@GetMapping("/users")
fun getUsers(): List<UserResponse> {
    return userService.getAll()  // 어떻게 JSON이 되는지 몰라도 됨
}
```

Ktor는 코드 흐름이 눈에 보이는 반면, Spring은 어노테이션이 뒤에서 처리하는 게 많아서 **"왜 되는지 모르고 쓰는 상태"** 가 오래 지속될 수 있습니다.

| | Spring + Kotlin | Ktor |
|--|----------------|------|
| Kotlin 문법 장벽 | 없음 | 없음 |
| 프레임워크 개념 장벽 | **높음** | 낮음 |
| 실무 활용도 | 매우 높음 | 중간 |
| 토이프로젝트 속도 | 느림 (설정 많음) | 빠름 |

---

**Q3. Spring 배워야 하는 개념들이 코틀린에도 있는 개념인가? Bean은 Java Bean 규약을 말하는 거야?**

**A3.**

이것들은 **Kotlin 언어 개념이 아니라 Spring 프레임워크 고유 개념**입니다.
다만 안드로이드에서 비슷한 걸 이미 써봤을 수 있습니다.

| Spring 개념 | 안드로이드 대응 | 차이 |
|------------|--------------|------|
| **IoC / DI** | Hilt / Koin | 거의 동일한 개념, 구현 방식만 다름 |
| **Bean 생명주기** | ViewModel 생명주기 | 비슷한 개념이지만 관리 주체가 다름 |
| **AOP** | 안드로이드에 없음 | Spring 고유, 처음 접하는 개념 |
| **JPA / Hibernate** | Room | 비슷한 ORM이지만 훨씬 복잡함 |
| **Spring Security** | 안드로이드에 없음 | 서버 전용 개념 |
| **트랜잭션** | Room의 @Transaction | 비슷하지만 범위가 훨씬 넓음 |
| **Auto Configuration** | 안드로이드에 없음 | Spring 고유 마법 |

**Bean은 Java Bean 규약과 전혀 다른 개념입니다.**

```
Java Bean 규약 (일반 Java 개념):
  - private 필드
  - getter/setter 있음
  - 기본 생성자 있음
  → 그냥 데이터 클래스 규약

Spring Bean (Spring 개념):
  - Spring IoC 컨테이너가 생성하고 관리하는 객체
  - 앱 실행 시 컨테이너가 알아서 만들어서 보관
  - 필요한 곳에 자동으로 주입해줌
```

안드로이드로 비유하면:

```kotlin
// Hilt - 안드로이드 DI
@HiltViewModel
class UserViewModel @Inject constructor(
    private val repository: UserRepository  // Hilt가 주입
) : ViewModel()

// Spring Bean - 서버 DI
@Service
class UserService(
    private val repository: UserRepository  // Spring이 주입
)
```

`@HiltViewModel`, `@Inject` 붙이면 Hilt가 객체 만들어서 관리하듯이,
`@Service`, `@Component` 붙이면 Spring이 그 객체를 **Bean**으로 등록해서 관리합니다.

**Bean = Spring 컨테이너가 관리하는 객체**

## 3. DB 설치 - PostgreSQL, MySQL 등 Docker로 띄우기

---

**Q1. DB는 서버 입장에서 안드로이드 개발자의 SQLite인가? SQLite를 ORM인 Room으로 접근하거나 직접 SQLite API로 DAO 만들어서 쓰듯이, 서버도 PostgreSQL/MySQL 같은 관계형 DB의 자체 API로 DAO 만들어서 직접 접근하거나 ORM 써서 추상화된 DSL로 접근할 수 있는 건가?**

**A1.**

정확하게 이해하셨습니다.

```
안드로이드                    서버
─────────────────────────────────────────
SQLite          ←→     PostgreSQL / MySQL
Room (ORM)      ←→     JPA/Hibernate, Exposed (ORM)
SQLiteDatabase  ←→     JDBC (직접 API)
DAO 인터페이스   ←→     Repository / DAO
```

완전히 동일한 구조입니다.

#### 직접 접근 (JDBC) vs ORM 비교

**JDBC - SQLiteDatabase API 직접 쓰는 것과 동일**
```kotlin
// 안드로이드 - SQLiteDatabase 직접
db.rawQuery("SELECT * FROM users WHERE id = ?", arrayOf(id.toString()))

// 서버 - JDBC 직접
connection.prepareStatement("SELECT * FROM users WHERE id = ?").apply {
    setLong(1, id)
    executeQuery()
}
```

**ORM - Room과 동일한 개념**
```kotlin
// 안드로이드 - Room
@Query("SELECT * FROM users WHERE id = :id")
fun getUser(id: Long): User

// 서버 - JPA (Spring)
interface UserRepository : JpaRepository<User, Long>
userRepository.findById(id)  // SQL 직접 안 써도 됨

// 서버 - Exposed (Ktor에서 많이 씀, Kotlin DSL)
Users.select { Users.id eq id }.single()
```

#### 서버 ORM 선택지

| ORM | 프레임워크 | 특징 |
|-----|---------|------|
| **JPA / Hibernate** | Spring | 자바 표준, 복잡하지만 강력 |
| **Exposed** | Ktor 친화적 | JetBrains 만든 Kotlin DSL, Room이랑 감각 비슷 |
| **jOOQ** | 둘 다 | SQL을 타입세이프 DSL로 작성 |

#### Ktor 쓴다면 Exposed 추천 이유

```kotlin
// Exposed - Kotlin DSL이라 안드로이드 개발자 감각이랑 가장 유사
object Users : Table() {
    val id = long("id").autoIncrement()
    val name = varchar("name", 50)
}

// 조회
Users.select { Users.id eq 1L }.map { it[Users.name] }

// 삽입
Users.insert { it[name] = "홍길동" }
```

Room의 `@Entity`, `@Dao` 구조랑 개념이 거의 같아서 진입장벽이 가장 낮습니다.

---

**Q2. JDBC는 어떤 DB를 쓰든 상관없이 추상화된 거야? 어디에 소속되어 있는 거야? 자바? 아님 관계형 DB에 포함된 프로토콜인가?**

**A2.**

**Java 소속입니다.** Java 표준 라이브러리(`java.sql`)에 포함된 **DB 접근 표준 인터페이스**입니다.

```
[ 내 코드 ]
    ↓  JDBC API 호출 (java.sql.*)
[ JDBC Interface ]  ← Java 표준 스펙 (인터페이스만 정의)
    ↓
[ JDBC Driver ]     ← 각 DB 회사가 구현체 제공
    ↓
[ PostgreSQL ]  /  [ MySQL ]  /  [ SQLite ]  /  [ Oracle ]
```

JDBC 자체는 **인터페이스(스펙)만 정의**하고, 실제 구현체는 각 DB 벤더가 드라이버 형태로 제공합니다.

```kotlin
// build.gradle - PostgreSQL 드라이버 추가
implementation("org.postgresql:postgresql:42.x.x")

// 코드 - JDBC API는 동일, 드라이버만 교체
val connection = DriverManager.getConnection(
    "jdbc:postgresql://localhost:5432/mydb",  // ← 여기만 바꾸면 MySQL, SQLite 등으로 교체 가능
    "username",
    "password"
)
```

드라이버만 교체하면 같은 JDBC 코드로 다른 DB를 쓸 수 있습니다.

추가로 JDBC는 **관계형 DB에 한정**됩니다. MongoDB 같은 NoSQL은 JDBC 안 씁니다.

```
관계형 DB  →  JDBC로 추상화 (PostgreSQL, MySQL, SQLite 등 다 지원)
NoSQL      →  각자 고유 방식 (MongoDB Driver, Redis Client 등)
```

Java가 워낙 서버 생태계에서 오래됐으니 웬만한 관계형 DB는 JDBC 드라이버를 당연히 제공합니다.

---

**Q3. 유명한 관계형 DB 몇 가지 예시와 장단점을 들어줘**

**A3.**

#### PostgreSQL
```
장점:
  - 기능이 가장 풍부 (JSON 타입, 배열, 고급 쿼리 등)
  - 표준 SQL 준수율 가장 높음
  - 대용량 / 복잡한 쿼리에 강함
  - 오픈소스, 무료
  - Supabase가 PostgreSQL 기반

단점:
  - MySQL보다 초기 설정이 약간 복잡
  - 단순 읽기 작업은 MySQL보다 살짝 느릴 수 있음
```

#### MySQL
```
장점:
  - 세계에서 가장 많이 쓰이는 관계형 DB
  - 레퍼런스, 자료가 압도적으로 많음
  - 단순 읽기 작업 빠름
  - 설정 쉬움

단점:
  - PostgreSQL보다 고급 기능 부족
  - Oracle이 인수 후 라이선스 이슈 (MariaDB로 포크된 이유)
```

#### MariaDB
```
장점:
  - MySQL과 거의 동일한 문법 (MySQL 포크)
  - 완전 오픈소스, 라이선스 자유로움
  - MySQL 대체재로 많이 쓰임

단점:
  - MySQL과 미묘한 차이가 있어 일부 호환 문제 발생 가능
  - PostgreSQL보다 고급 기능 부족한 건 동일
```

#### SQLite
```
장점:
  - 파일 하나가 DB (설치 불필요)
  - 안드로이드 로컬 DB와 동일
  - 개발/테스트 환경에 적합

단점:
  - 동시 접속 처리가 매우 약함
  - 실서버 프로덕션에는 부적합
```

#### 한눈에 비교

| | PostgreSQL | MySQL | MariaDB | SQLite |
|--|-----------|-------|---------|--------|
| **성능** | 복잡한 쿼리 강함 | 단순 읽기 강함 | MySQL과 유사 | 단일 사용자용 |
| **기능** | 최고 | 중간 | 중간 | 최소 |
| **자료량** | 많음 | 압도적 | 많음 | 많음 |
| **라이선스** | 완전 무료 | Oracle 소유 | 완전 무료 | 완전 무료 |
| **용도** | 실무 프로덕션 | 실무 프로덕션 | 실무 프로덕션 | 로컬/테스트 |

**토이프로젝트 추천: PostgreSQL** - Supabase도 PostgreSQL 기반이고, 기능이 가장 풍부하며 실무에서도 많이 씁니다.
